# OpenSpec 的用途

OpenSpec 是一种用于“规范先行”开发的工作方式：先写清楚需求和约束，再进入实现阶段。它的核心价值是把“想做什么、为什么做、怎么验收”在编码前对齐清楚，减少返工和沟通成本。

## GitHub 地址

- 官方仓库：https://github.com/Fission-AI/OpenSpec

## 主要用途

- 明确需求边界：把功能目标、输入输出、限制条件写成可讨论的规范。
- 统一团队认知：产品、设计、开发、测试基于同一份文档协作。
- 提前发现风险：在实现前暴露冲突、遗漏和不合理假设。
- 指导实现与测试：将规范中的验收标准直接映射为测试点。
- 便于变更管理：需求变化时，先更新规范，再追踪实现影响范围。

## 适用场景

- 新功能立项：需要在多角色之间快速达成一致。
- 复杂改造：涉及多个模块，容易出现隐性耦合和回归风险。
- 对质量要求高的项目：需要可追溯的设计决策与验收依据。
- AI 辅助开发：让模型基于明确规范生成或修改代码，降低偏差。

## 实践建议

1. 先写“目标与非目标”，避免范围失控。
2. 为关键流程定义可验证的验收标准。
3. 规范更新要与代码变更保持同步。
4. 在评审中优先讨论规范，再进入实现细节。

简而言之，OpenSpec 的用途是把“口头需求”变成“可执行、可验证、可演进”的工程规范，让开发过程更稳定、更可控。
